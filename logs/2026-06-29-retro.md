*   **核心痛点**：在 Assistant 对话中通过指令快捷触发 QC 工作流（`qcrun`）后，大模型会输出一个包含 `WorkflowProgressCard` 的进度消息。然而点击卡片右下角的 **`[在工作台查看本次 Run]`** 动作按钮，虽然视图切入到了 `'workbench'` (任务列表)，但工作台却无法精准渲染刚才启动的运行数据，展现出一片空白或旧日期数据，破坏了极速联动体验。

## 🧬 三、 架构品味与设计决策 (Architecture & Design Decoupling)

全栈 AI 工程师绝不堆砌垃圾代码。本日我们做出了一次经典的**异步拉取自适应对齐（Asynchronous Pull Alignment）**架构抉择：

*   **Before (原始混沌状态)**：
    *   在 `WorkflowProgressCard` 中，按钮被点击时，大模型在底层只回调了一个单纯的 `qcRunId`。
    *   而父组件 `onOpenWorkbench` 的回调只期望通过 `batchRunId` 或 `date` 来对齐当前项目的 `selectedProjectId` 和 `workbenchFocusDate`。对于单个 `qcRunId` 传入的场景（即聊天卡片点击），两个参数均为 `undefined`。
    *   这导致跳转虽然发生，但没有任何业务日期和项目 ID 的聚焦更新，系统退化为加载“今日”空面板，联动彻底失效。
*   **After (优雅自愈对齐)**：
    *   **物理自愈拦截**：我们在 `onOpenWorkbench` 的回调最前置，注入了一个单个 `_runId` 的**确定性拦截器**。
    *   当发现仅有单 Run ID 传入时，前置异步发起对 `/query/fund-qc/runs/${_runId}/detail` 的拉取请求。
    *   在毫秒级内反向解析出该 Run 对应的 `projectId`、`businessDate`（如 2026-06-27），并动态设置这些状态，随后再切入 workbench 视图。
*   **第一性原理支撑**：
    *   大模型在界面上产生的点击回调是**概率性、不完整的（只含 Run ID）**；
    *   我们不强求 AI 在每次点击时都精准携带庞杂的上下文，而是用**确定性的异步拉取机制（api.getFundQcRunDetail）**在后台补齐缺损的上下文，完美隔离了概率性。
