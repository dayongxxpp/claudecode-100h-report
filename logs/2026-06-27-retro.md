## 🚀 一、 场景演练与实用主义重构对照 (Scenarios & Before-After)

### 📂 【Before ➔ After】VFS 路径逃逸防御重构

*   **痛点 (Antipattern)**：
    *   在开发 `feat/agent-vfs-phase1` 时，原始逻辑直接将用户的绝对/相对路径传给 `FileSystemSandbox`。
    *   *AI 的概率性失误*：当用户输入复杂或恶意的绝对路径时，AI 可能会计算出一个溢出沙盒根目录的路径，造成安全红线失守。
*   **重构方案 (Pattern)**：
    *   不在 Prompt 里反复哀求 AI *“请不要越界”*。
    *   直接在 `PolicyGuardrailService` 和 `RegulationMountHandler` 中手写**确定性防逃逸拦截**：
    ```typescript
    // 1. 物理清洗 relativePath，强制去除前后首斜杠，阻断 "../" 相对溢出
    const rel = payload.relativePath.replace(/^\/+|\/+$/g, '');
    const sandboxRelPath = rel ? `paf/regulations/${rel}` : 'paf/regulations';
    
    // 2. 物理边界检查：确保未跳出 regs 根目录
    if (!mdSavePath.startsWith(regDir + path.sep)) {
      throw new Error(`regulations path escaped root: ${sourceFileName}`);
    }
    ```
*   **实用主义收获**：**“用确定性的代码写死校验逻辑，只把高成本的认知工作交给 AI（概率），把低成本的安全防线交给自己（确定性）。”**

## ❌ 二、 失败模式深度剖析 (Failure Modes)

### 🛠️ 【反模式】重名类名与文件静默覆盖冲突
*   **失败现象**：在设计 Outbound 质控日志同步时，AI 尝试在 `valuation-qc/skill/` 目录下创建一个名为 `RegulationSyncListener` 的监听器，浑然不知当前分支上已经存在一个同名、负责 VFS 同步的监听器。
*   **物理成因**：由于没有进行 Git Status 和本地文件树的预检，AI 的知识库停留在了 main 分支的状态，产生了**局部上下文黑洞**。
*   **纠偏行动**：
    *   **不要让 AI 直接写代码，先让它运行命令自检**。通过 `git status` 预检，主动将新监听器改名为 `OpenChronicleDailySyncListener`，并落盘至 `openchronicle-daily-sync.listener.ts`，物理错开开发热区。

## ⚖️ 三、 与 Cursor/Copilot CLI 对比 (Utility博弈)

*   **Claude Code (CLI)**：极度纯粹。通过 `--dangerously-skip-permissions` 后，适合无人值守的闭环开发。一句话就能自主读取 `#233` PR 的 GitHub comments，自主编写 Handler，并就地通过 `pnpm test` 验证。
*   **Cursor (GUI)**：虽然在大段代码的重构和多文件跳转阅读上拥有良好的视觉体验，但在自动跑单测、自建 VFS 代码框架时，操作摩擦和心智开销明显高于一键式 CLI 闭环。

## 🏦 四、 金融场景最佳实践 (Strict Privacy Gotchas)

### 🔒 【第一性原理】Strict Session-Only 会话沙盒隔离
*   **痛点**：金融审计严格禁止在后台服务中加载录屏、前台窗口监控和 OCR 文字捕捉模块，防范数据外泄红线。
*   **解法**：
    *   彻底**切断一切前台屏幕和 OCR 监控**（禁用 `memory_s1_events` 扫描）。
    *   将 OpenChronicle 权限收缩在**本助理内部的历史对话 Sessions 内**，通过 `OPEN_READONLY` 方式只读调阅 `memory_sessions` 表。
    *   仅凭对历史对话结论和对账摘要的回溯，就能优雅地还原上下文情境，实现 100% 物理合规！
